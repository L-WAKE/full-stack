## 2. 什么是自动化测试

自动化测试，就是把原来需要人工重复操作的验证动作，交给程序自动执行。

例如，人工测试房源管理系统时，通常会做这些事：

1. 启动后端服务
2. 打开登录页
3. 输入管理员账号密码
4. 进入房源列表
5. 新增一条房源
6. 检查新增结果
7. 删除测试数据

如果这些动作每次发版、每次改代码、每次修 bug 都要人工重复，就会非常耗时，而且容易漏测。  
自动化测试的作用，就是把这些步骤写成脚本或测试代码，让机器自动完成并给出结果。

---

## 3. 为什么房源管理系统特别适合做自动化测试

房源管理系统属于典型的后台业务系统，特点是：

- 表单多
- 列表多
- 增删改查多
- 权限控制多
- 业务规则多

这类系统非常容易出现以下问题：

- 登录改了，别的页面进不去了
- 房源新增字段变更后，编辑接口出错
- 状态流转逻辑改了，列表展示不一致
- 菜单权限改了，按钮消失或越权
- 一个模块修复后，另一个模块回归失败

自动化测试最擅长解决的，就是这种“改一点，影响很多”的问题。

## 4. 自动化测试的核心价值

对于当前房源管理系统，自动化测试至少能带来 5 个直接收益：

### 4.1 提前发现回归问题

开发改动后，测试脚本立即帮你验证登录、房源管理、工单管理等核心能力是否还正常。

### 4.2 降低人工回归成本

以前每次都要手工点十几二十分钟，以后只需要运行一条命令。

### 4.3 让需求迭代更安心

当你后续继续增加合同管理、账单管理、导入导出等模块时，已有自动化测试会像安全网一样保护老功能。

### 4.4 帮助定位问题

测试失败时，通常可以快速看出是：

- 服务没起来
- 登录失效
- 某个接口返回异常
- 某个前端交互失效

### 4.5 为持续集成打基础

后续如果接入 GitHub Actions、Jenkins 或 GitLab CI，每次提交代码后都能自动校验质量。

## 5. 自动化测试不是只有一种

很多初学者一提自动化测试，就只想到“浏览器录制操作”。  
其实自动化测试是分层的，不同层解决不同问题。

最推荐的结构是“测试金字塔”。

### 5.1 单元测试

单元测试关注最小可测试单元，通常是：

- 一个方法
- 一个函数
- 一段校验逻辑
- 一个状态转换规则

适合验证：

- 房源状态枚举转换是否正确
- 参数校验逻辑是否符合预期
- 数据转换逻辑是否正确

特点：

- 速度快
- 维护成本低
- 定位问题准

### 5.2 接口测试

接口测试关注前后端之间的业务接口是否正常。

适合验证：

- 登录接口是否返回 token
- 房源列表查询是否正确分页
- 新增房源是否成功写入数据
- 删除房源是否正确返回成功结果
- 无权限访问时是否被拦截

特点：

- 性价比高
- 能覆盖大量核心业务
- 比 UI 自动化稳定很多

### 5.3 UI 自动化测试

UI 自动化测试关注真实用户操作流程。

适合验证：

- 登录页面是否能正常登录
- 房源新增弹窗是否可打开并保存
- 查询条件输入后列表是否刷新
- 删除确认框是否正确弹出

特点：

- 最接近真实用户
- 但执行最慢、维护成本最高

### 5.4 冒烟测试

冒烟测试是一种“先看看系统活没活”的快速验证。

它不追求覆盖所有细节，而是优先验证系统主链路是否能跑通。  
你当前仓库里已经有现成例子：`tools/smoke-test-house-service.ps1`。

### 第二阶段：补后端接口测试

目标：用较低成本保护核心业务逻辑。

优先覆盖：

- 房源管理接口
- 业主管理接口
- 租客管理接口
- 维修工单接口
- 保洁工单接口

### 第三阶段：补前端组件/页面测试

目标：验证关键页面行为，不依赖人工点击。

优先覆盖：

- 登录页
- 房源列表页

- 管理员登录并进入看板
- 新增房源并查询到结果
- 新增维修工单并修改状态

目标：每次提交代码自动运行测试。

## 8. 自动化测试的标准流程

下面是比较推荐、也最适合教学和实操的一套流程。

### 8.1 第一步：明确测试目标

不要一开始就想着“把系统所有页面都自动化”。

应该先确定：

- 哪些模块最重要
- 哪些功能最容易出问题
- 哪些流程最适合自动化

对本项目来说，优先级建议如下：

1. 登录与鉴权
2. 房源管理
3. 工单管理
4. 业主/租客管理
5. 菜单权限

### 8.2 第二步：梳理测试场景

每个模块都要先列出典型场景，再写脚本。

以房源管理为例，至少应该有这些场景：

- 正常新增房源
- 缺少必填项时新增失败
- 编辑房源成功
- 删除房源成功
- 按关键字查询房源
- 按状态过滤房源
- 修改房源状态成功

### 8.3 第三步：选择测试层级

不是所有场景都要用 UI 测试。

更合理的做法是：

- 参数校验：优先接口测试或单元测试
- 业务逻辑：优先接口测试
- 页面主流程：用 UI 自动化

### 8.4 第四步：准备测试环境

测试运行前，必须保证：

- 数据库可用
- Redis 可用
- 后端服务可启动
- 前端服务可访问
- 测试账号存在

### 8.5 第五步：准备测试数据

测试数据应该满足以下要求：

- 可控
- 可重复
- 能清理
- 不污染正式数据

建议给自动化测试数据统一加前缀，例如：

- `AUTO_`
- `SMOKE_`
- `E2E_`

这样后续排查和清理都很方便。

### 8.6 第六步：编写测试脚本

每条测试脚本通常包含以下结构：

1. 前置准备
2. 执行操作
3. 断言结果
4. 清理数据

例如：

1. 登录拿到 token
2. 调用新增房源接口
3. 断言返回成功并校验返回字段
4. 删除刚才新增的测试房源

### 8.7 第七步：执行与分析结果

执行后要关注的不只是“有没有报错”，还要看：

- 哪一步失败
- 是环境问题还是业务问题
- 是接口异常还是断言错误
- 是否出现偶发性失败

### 8.8 第八步：纳入日常开发流程

自动化测试真正有价值，不是写出来放着，而是要进入日常流程：

- 本地开发后先跑一次
- 提交代码前跑一次
- 合并分支前自动跑一次
- 发布前自动跑一次

## 9. 编写自动化测试必须遵守的规则

这是最关键的一部分。  
如果不遵守这些规则，测试脚本很快就会变成负担。

### 9.1 测试必须可重复执行

同一条测试脚本，不管今天跑还是明天跑，都应该得到稳定结果。

不要依赖：

- “数据库里刚好有一条数据”
- “我刚刚手动点过一次”
- “这个账号昨天还能用”

### 9.2 测试数据要自给自足

测试开始前自己创建数据，测试结束后自己清理数据。  
不要把成功依赖建立在人工准备上。

### 9.3 一条测试只验证一个核心目标

例如：

- 一条测试专门验证“登录成功”
- 一条测试专门验证“缺失房源名称时新增失败”

不要在一条测试里同时验证五六件事情，否则一旦失败就很难排查。

### 9.4 断言要明确

不要只判断“接口返回了 200”。

还应该检查：

- 业务返回码
- 返回消息
- 返回字段
- 数据是否真实写入
- 状态是否真的变化

### 9.5 尽量避免脆弱选择器

做前端 UI 自动化时，不要过度依赖：

- 深层 CSS 结构
- 易变化的 class 名
- 文案中容易改动的内容

更推荐使用：

- `data-testid`
- 稳定的按钮文本
- 稳定的表单标识

### 9.6 UI 自动化只测关键主流程

UI 自动化成本最高，不要把每一个小弹窗、小样式都做成 E2E。

最推荐覆盖：

- 登录
- 新增
- 编辑
- 删除
- 查询

### 9.7 测试环境与开发环境要分离

最好有独立测试库，至少不要直接拿正式环境做自动化测试。

### 9.8 每修一个 bug，补一条回归测试

这是自动化测试长期产生价值的关键习惯。

如果某个 bug 修复过，但没有补测试，未来它还会再回来。

### 9.9 失败日志要可读

当测试失败时，最好能快速看出：

- 请求参数是什么
- 接口返回了什么
- 页面停在哪一步
- 哪条断言没通过

### 9.10 测试命名要清晰

测试名称应该能直接表达行为和预期，例如：

- `should_login_success_with_admin_account`
- `should_reject_create_house_when_house_name_missing`
- `should_update_house_status_to_occupied`

## 13. 冒烟测试、接口测试、UI 测试分别怎么理解

这里用最通俗的话讲一下三者的区别。

### 13.1 冒烟测试

关注：系统活着没，主链路通不通。

比如：

- 能不能登录
- 核心接口能不能调用

### 13.2 接口测试

关注：业务规则对不对。

比如：

- 新增房源时缺字段是否失败
- 修改状态是否真的成功
- 删除后列表是否消失

### 13.3 UI 测试

关注：用户操作路径是否正常。

比如：

- 页面有没有渲染出来
- 按钮点了有没有反应
- 表单提交后有没有结果反馈

简单理解就是：

- 冒烟测试保“能跑”
- 接口测试保“业务正确”
- UI 测试保“用户可用”

## 14. 自动化测试常见误区

### 14.1 一开始就想全量自动化

结果通常是写不完、维护不动、最后放弃。

正确做法是先抓核心主流程。

### 14.2 过度依赖 UI 自动化

UI 自动化最贵、最慢、最容易碎。  
很多逻辑更适合放在接口测试里验证。

### 14.3 没有测试数据管理

今天能跑，明天跑不了，通常就是因为测试数据不独立、不清理。

### 14.4 测试断言太弱

只断言“没报错”是不够的。  
一定要断言关键字段和业务结果。

### 14.5 只写测试，不纳入流程

如果测试从不在提交前或发布前执行，它的价值会大幅下降。

## 15. 如何判断自动化测试是否做得好

一套自动化测试做得好，通常有这些表现：

- 新人也能按文档跑起来
- 本地运行稳定，不容易偶发失败
- 失败后能快速定位原因
- 核心业务改动后能及时报警
- 测试代码本身不难维护

反过来说，如果测试经常：

- 今天过、明天挂
- 一挂就看不懂
- 改个文案就全红
- 跑一次要半小时

那就说明测试设计需要优化。

## 16. 给当前项目的一套实践建议

综合当前仓库情况，最推荐的推进顺序如下：

1. 保留并持续使用现有 PowerShell 冒烟脚本

6. 最后把测试接入 CI

这样做的好处是：

- 成本低
- 起步快
- 学习曲线平滑
- 每一步都能看到成果

## 17. 学习与实操建议

如果你是第一次系统性接触自动化测试，不建议同时学太多工具。  
最好的学习方式是“边改项目边测试”。

推荐节奏：

1. 先理解现有冒烟脚本做了什么
2. 手动跑通一次
3. 自己尝试给房源模块补一段类似脚本
4. 再进入正式测试框架
5. 每学一个知识点，就在当前项目里落地一条测试

这样你学到的不是抽象概念，而是真正能用在项目里的能力。
