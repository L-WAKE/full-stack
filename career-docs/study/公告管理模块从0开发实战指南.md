# 公告管理模块从 0 开发实战指南

## 1. 这份文档是做什么的

这是一份专门给前端开发者看的“后端入门 + 全栈落地”实战文档。目标不是只讲概念，而是带你在当前这个房源管理系统项目里，完整做出一个“公告管理”模块。

你会在这个过程中真正理解：

- 一个后台模块为什么要先设计表结构
- 后端的 `Controller / Service / Mapper / Entity` 各自负责什么
- 前端页面为什么依赖后端接口约定
- 一个 CRUD 模块从数据库到页面是怎么串起来的
- 以后你自己做“活动管理”“合同模板管理”“消息中心”等模块时，如何复用同一套方法

## 2. 当前项目技术栈和目录

当前仓库里，和房源管理系统有关的代码主要在这两个目录：

- 后端：`backend/house-service`

### 2.1 后端技术栈

- `Spring Boot 3`

- `Spring Security`

### 2.2 前端技术栈

- `Vue 3`

- `Pinia`
- `Vue Router`
- `Axios`

### 2.3 当前项目里一个标准模块的结构

以后你做任何后台模块，都可以先在脑子里套这个结构：

1. 数据库先有一张表
2. Java 里用 `Entity` 映射这张表
3. `Mapper` 负责最基础的数据读写
4. `Service` 负责业务逻辑
5. `Controller` 负责暴露接口
6. 前端 `api/*.js` 负责调接口
7. 前端 `views/*` 负责页面展示和交互

你可以把它理解成：

- 数据库 = 仓库
- Entity = 仓库货架标签
- Mapper = 仓库搬运工
- Service = 业务经理
- Controller = 前台窗口
- 前端页面 = 用户操作台

## 3. 为什么“公告管理”很适合练后端

“公告管理”是一个很好的入门模块，因为它：

- 业务简单，不容易被复杂规则绊住
- 又是完整 CRUD，能把后端核心流程全部练到
- 同时还能加一点真实业务，比如发布状态、置顶、发布时间

建议我们这次做一个“后台管理端公告模块”，面向房源管理系统内部员工使用。

## 4. 先定义清楚：这个公告模块要做什么

### 4.1 模块目标

在房源管理系统后台增加一个“公告管理”菜单，支持：

- 查看公告列表
- 按标题、状态筛选
- 新增公告
- 编辑公告
- 删除公告
- 发布 / 下线公告
- 置顶 / 取消置顶
- 查看详情

### 4.2 公告字段建议

我们先做一个“够用、标准、适合练手”的版本，字段如下：

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| id | 主键 | 1 |
| org_id | 组织 ID | 1 |
| title | 公告标题 | 五一假期值班安排 |
| content | 公告内容 | 请各门店按排班执行 |
| summary | 公告摘要 | 五一值班安排已发布 |
| status | 状态 | DRAFT / PUBLISHED / OFFLINE |
| pinned | 是否置顶 | 0 / 1 |
| publish_time | 发布时间 | 2026-04-14 10:00:00 |
| created_by | 创建人 | 1 |
| created_at | 创建时间 | 自动维护 |
| updated_by | 更新人 | 1 |
| updated_at | 更新时间 | 自动维护 |
| deleted | 逻辑删除 | 0 / 1 |

### 4.3 为什么这样设计

- `title`：列表和详情都会展示，是最基础字段
- `summary`：列表页一般不适合直接展示正文，所以保留摘要
- `content`：正文内容
- `status`：区分草稿、已发布、已下线
- `pinned`：后台列表经常需要置顶逻辑
- `publish_time`：发布动作发生后，要有明确时间
- `deleted`：当前项目已经使用逻辑删除，所以继续沿用

## 5. 从 0 开发一个后端模块，标准顺序是什么

后端开发不要一上来就写 Controller。正确顺序应该是：

1. 先想清楚业务字段和状态
2. 设计数据库表
3. 设计接口
4. 写 Entity
5. 写 Mapper
6. 写 Service
7. 写 Controller
8. 用接口工具或前端页面验证

很多前端同学学后端容易乱，是因为顺序反了。  
如果你先写页面，再倒推接口，再倒推数据库，就会一直觉得后端很乱。

## 6. 公告管理模块的完整开发路线

下面这套路线，就是我们接下来要一步一步完成的。

### 第 1 步：明确业务规则

先定规则，不急着写代码。

这次我们约定：

- 新建公告默认是 `DRAFT`
- 发布后状态变成 `PUBLISHED`
- 下线后状态变成 `OFFLINE`
- 只有 `PUBLISHED` 状态才有 `publishTime`
- 置顶只是排序能力，不影响状态
- 删除采用逻辑删除

### 第 2 步：设计数据库表

在 `backend/house-service/src/main/resources/schema.sql` 里新增 `notice` 表。

这是后端最关键的一步。  
因为后面的 Entity、接口返回结构、前端表单字段，几乎都要围着这张表转。

### 第 3 步：创建后端模块目录

建议新增目录：

- `backend/house-service/src/main/java/com/example/house/modules/notice`
- `backend/house-service/src/main/java/com/example/house/modules/notice/entity`
- `backend/house-service/src/main/java/com/example/house/modules/notice/mapper`

### 第 4 步：写 Entity

新增 `NoticeEntity`，把表字段映射到 Java 类。

这一层的本质是：

- 表里叫 `publish_time`
- Java 里写成 `publishTime`

MyBatis-Plus 会帮你完成大部分映射。

### 第 5 步：写 Mapper

新增 `NoticeMapper extends BaseMapper<NoticeEntity>`。

这一层通常很薄，因为当前项目使用了 MyBatis-Plus，很多基础 CRUD 已经帮你做好了。

### 第 6 步：写 Service

这里是核心。

你要在 `NoticeService` 里处理：

- 分页查询
- 详情查询
- 新增
- 编辑
- 删除
- 发布
- 下线
- 置顶 / 取消置顶

同时，Service 里还要做一些业务判断，比如：

- 不存在的公告不能操作
- 已删除的数据不能查到
- 发布时自动写入 `publishTime`
- 下线时状态改为 `OFFLINE`

### 第 7 步：写 Controller

在 `NoticeController` 暴露接口，例如：

- `GET /api/notices`
- `GET /api/notices/{id}`
- `POST /api/notices`
- `PUT /api/notices/{id}`
- `DELETE /api/notices/{id}`
- `PUT /api/notices/{id}/publish`
- `PUT /api/notices/{id}/offline`
- `PUT /api/notices/{id}/pin`

### 第 8 步：补初始化菜单数据

当前项目菜单来自后端数据初始化。  
所以要在 `DataInitializer` 中补：

- 一级菜单：如果你想挂在“系统管理”下，则不需要新增一级
- 二级菜单：`/system/notice`
- 权限码：例如 `notice:view`

### 第 9 步：前端新增 API 封装

在 `frontend/house-admin/src/api/resources.js` 增加：

- `getNotices`
- `getNoticeDetail`
- `createNotice`
- `updateNotice`
- `deleteNotice`
- `publishNotice`
- `offlineNotice`
- `toggleNoticePin`

### 第 10 步：前端增加路由

在 `frontend/house-admin/src/router/index.js` 中增加：

- `system/notice`

对应页面组件：

- `frontend/house-admin/src/views/system/NoticeView.vue`

### 第 11 步：前端实现列表页 + 弹窗表单

建议页面包含：

- 筛选区：标题关键字、状态
- 表格区：标题、状态、是否置顶、发布时间、更新时间
- 操作列：编辑、删除、发布、下线、置顶
- 弹窗表单：标题、摘要、内容、状态

### 第 12 步：联调和验证

最终要验证这些场景：

- 新建公告成功
- 列表可看到新数据
- 编辑后数据更新
- 发布后状态变成已发布，发布时间自动写入
- 下线后状态变成已下线
- 置顶后排序靠前
- 删除后列表不再展示

## 8. 数据库表设计建议

建议新增表：

```sql
CREATE TABLE IF NOT EXISTS notice (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    org_id BIGINT DEFAULT 1,
    title VARCHAR(128) NOT NULL,
    summary VARCHAR(255),
    content TEXT NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'DRAFT',
    pinned TINYINT NOT NULL DEFAULT 0,
    publish_time TIMESTAMP NULL,
    created_by BIGINT DEFAULT 1,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_by BIGINT DEFAULT 1,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted TINYINT NOT NULL DEFAULT 0
);
```

### 8.1 字段说明

- `title` 用 `VARCHAR(128)`，一般够用了
- `summary` 用 `VARCHAR(255)`，足够当列表摘要
- `content` 用 `TEXT`，因为正文长度可能比较大
- `status` 存字符串，可读性更好
- `pinned` 用 `TINYINT`
- `publish_time` 允许为空，因为草稿没有发布时间

### 8.2 列表排序建议

公告列表建议按下面规则排序：

1. `pinned` 倒序
2. `publish_time` 倒序
3. `id` 倒序

这样用户会感觉更像真实后台系统。

## 9. 后端代码应该怎么组织

### 9.1 Entity 层负责什么

它只做一件事：映射数据库字段。

不要在 Entity 里写复杂业务逻辑。

### 9.2 Mapper 层负责什么

它负责最底层数据访问。

在当前项目里，多数时候你只需要：

```java
@Mapper
public interface NoticeMapper extends BaseMapper<NoticeEntity> {
}
```

### 9.3 Service 层负责什么

这是后端的“脑子”。

你所有“业务规则”都应该尽量放这里，例如：

- 如果标题为空，不允许创建
- 如果公告不存在，抛异常
- 发布时自动写发布时间
- 置顶时更新 pinned 字段

### 9.4 Controller 层负责什么

它只是 HTTP 入口。

可以理解为：

- 接收前端请求
- 校验参数
- 调用 Service
- 返回统一响应结构

它不应该承载太多业务细节。

### 10.1 一个标准管理页的组成

你可以继续沿用当前项目已有页面模式：

- 页面标题
- 查询表单
- 表格
- 分页
- 新增 / 编辑弹窗

### 10.2 公告页面建议字段

表格列建议：

- 标题
- 摘要
- 状态
- 是否置顶
- 发布时间
- 更新时间
- 操作

弹窗表单建议字段：

- 标题
- 摘要
- 正文
- 状态
- 是否置顶

### 10.3 为什么前端也要理解状态流转

如果你不理解后端状态流转，就很容易在前端写出奇怪交互。

例如：

- 草稿才能发布
- 已发布才能下线
- 已下线可以重新编辑再发布

前端按钮显隐、禁用状态，都依赖这些业务规则。

## 11. 对前端开发者来说，后端最容易卡住的点

### 11.1 误区一：觉得后端就是写接口

不是。  
后端的核心是：数据结构 + 业务规则 + 数据一致性。

接口只是出口。

### 11.2 误区二：先写页面再倒逼后端

这样会经常把自己绕晕。  
更好的顺序是：先表、再接口、再后端、再前端。

### 11.3 误区三：把 Service 写成“中转站”

很多新手会写成：

- Controller 接请求
- Service 原样调用 Mapper
- 没有任何业务判断

这样看似分层了，其实没有真正理解业务层。

### 11.4 误区四：不做状态设计

像公告这种模块，如果没有 `DRAFT / PUBLISHED / OFFLINE`，后面很快会乱。

所以你在写表之前，就要先想状态。

## 15. 学习建议

你已经有 6 年前端经验，这其实是巨大优势，不是劣势。

你缺的往往不是编码能力，而是“后端分层思维”和“数据建模习惯”。

你可以这样学：

- 每写一个后端类，先问自己“它负责哪一层”
- 每加一个字段，先问自己“这个字段是展示字段，还是业务字段”
- 每加一个按钮，先问自己“它本质上改变了什么状态”

只要把这三个问题练熟，你上手后端会比你想象中快很多。
