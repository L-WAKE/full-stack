# 交付清单

## 总原则

按阶段稳定推进。
不要一上来追求大而全的 Agent。
先把 RAG 主链路做完整。

## 第 1 阶段：基础设施

要完成：

- 前端项目初始化
- FastAPI 项目初始化
- PostgreSQL 连接与迁移
- Redis 连接
- Qdrant 连接
- 登录和基础鉴权

验收标准：

- 前后端都能在本地启动
- 用户可以登录
- 后端能连通 PostgreSQL、Redis、Qdrant
- 环境变量和基础配置已经落到代码或示例配置文件中

## 第 2 阶段：知识库空间与文档中心

要完成：

- 知识库空间增删改查
- 空间成员分配
- 文档上传
- 文档列表
- 解析状态展示

验收标准：

- 用户可以创建和编辑知识库空间
- 用户可以上传 PDF 和 DOCX
- 上传后文档状态进入 `PENDING`
- 前端能看到解析状态和失败原因占位

## 第 3 阶段：解析、切片、向量化

要完成：

- PDF 解析
- DOCX 解析
- 结构化清洗
- 语义切片
- 向量生成
- Qdrant 索引

验收标准：

- 解析后的文本被持久化
- PostgreSQL 中有切片元数据
- 已保存向量映射关系
- 成功文档状态变为 `READY`
- 失败文档状态变为 `FAILED`

## 第 4 阶段：检索与聊天

要完成：

- 检索接口
- 聊天接口
- SSE 流式输出
- 引用来源返回
- 会话管理

验收标准：

- 用户可以在指定空间内发起提问
- 系统返回流式回答
- 系统返回带文档名和页码的引用信息
- 用户可以查看历史会话和消息

## 第 5 阶段：权限与优化

要完成：

- 空间访问限制
- 文档可见范围限制
- 检索范围限制
- 重排或质量优化
- Docker Compose 部署

验收标准：

- 无权限用户不能检索到受限内容
- 检索不会跨越无权限空间
- 本地多服务部署可运行
- 项目能被完整讲清楚，链路闭环明确

## 六周节奏建议

如果用户要一个可执行的个人开发排期，按下面节奏推进：

1. 第 1 周：前后端初始化、登录、知识库空间
2. 第 2 周：上传、文档列表、文件存储
3. 第 3 周：PDF/DOCX 解析与切片
4. 第 4 周：向量化和 Qdrant 检索
5. 第 5 周：聊天页、SSE、引用来源
6. 第 6 周：权限隔离、调优、Docker 部署

## 首期完成定义

只有下面全部成立，才能认为第一阶段完成：

- PDF 和 DOCX 上传可用
- 文档可以异步解析和切片
- 向量已存储且可检索
- 回答基于检索上下文生成
- 引用来源已返回并展示
- 会话按知识库空间存储
- SSE 流式输出可用
- 检索时权限已生效
- Docker Compose 可在本地运行

## 继续开发规则

当用户说“继续开发”时：

1. 先判断当前代码已经完成到哪个阶段
2. 以代码实际完成度为准，不以文档为准
3. 从第一个未满足验收条件的阶段继续推进
