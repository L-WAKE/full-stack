# MySQL 最佳实践

## 作用

当你在设计表结构、检查 SQL、优化查询性能或排查慢 SQL 时，使用这个 skill。

## 适用场景

- 表设计和索引设计
- SQL 审查与性能优化
- 分页方案设计
- 唯一性与一致性约束设计
- 大表和慢查询问题排查

## 核心规则

1. 索引必须围绕真实查询场景设计，而不是凭感觉添加。
2. 应用层查询尽量避免 `select *`，除非表极小且使用场景非常明确。
3. 高频查询应优先保证筛选条件明确、排序稳定、访问路径可预测。
4. 联合索引顺序要尽量匹配常见的筛选和排序方式。
5. 大数据量下要谨慎使用 `offset` 分页，性能敏感场景优先考虑游标分页或基于主键的翻页。
6. 唯一性约束尽量由数据库直接保证，而不是只依赖应用层判断。
7. 写事务路径尽量短、尽量稳定。
8. 可空字段、默认值、长文本字段要和真实业务需求匹配，不要随意放宽。
9. 对函数包裹索引列、隐式类型转换、无边界模糊查询等容易触发全表扫描的写法保持警惕。
10. 大表变更要考虑锁表风险、迁移方式和上线策略。

## 查询建议

- 只查需要的字段。
- 尽量使用可命中索引的过滤条件。
- 高频路径中不要对索引列做函数处理。
- 批量场景优先批量查询，避免循环单查。
- 多表关联时要关注连接后的数据量膨胀问题。

## 索引建议

- 只有在真实读取场景或约束需要下才建立索引。
- 避免重复索引和意义接近的冗余索引。
- 给高频写入表增加多个二级索引前，要评估写放大成本。
- 联合索引要结合查询最左匹配原则和排序需求一起设计。

## 分页建议

- 小数据量或后台管理场景可以接受 `offset` 分页。
- 大数据量、高并发、严格性能要求场景优先考虑游标分页或 seek 分页。
- 排序条件要稳定，最好带唯一字段作为兜底排序。

## 数据建模建议

- 数值、时间、字符串类型要根据业务语义选择，不要图省事全部用大而泛的类型。
- 时间字段要统一时区策略。
- 软删除只有在业务确实需要时才使用，并且要评估查询复杂度和索引影响。
- 审计字段要按需要增加，不要机械式堆字段。

## 使用时的检查清单

使用这个 skill 时，重点检查：

- 这条 SQL 是否对应了真实存在且合理的索引
- 当前分页方案是否适合表规模
- 唯一性和完整性是否由 MySQL 本身保证
- 这个改动是否会带来明显的写压力或维护成本
- 数据量继续增长后，这条 SQL 会不会明显退化
- 当前表结构变更是否适合生产环境执行

## 期望结果

- 查询更快、更稳定
- 索引质量更高
- 表结构演进更安全
- 生产环境慢 SQL 风险更低
