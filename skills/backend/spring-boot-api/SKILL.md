# Spring Boot 接口开发规范

## 作用

当你在基于 Spring Boot、MySQL、Redis 的后端服务中开发或评审接口时，使用这个 skill。

## 适用场景

- 设计 REST 接口
- 重构 Controller、Service、Repository 之间的调用关系
- 增加参数校验、事务控制、异常处理
- 检查 DTO、Entity、持久层边界
- 提升日志质量、代码一致性和可维护性

## 核心规则

1. `Controller` 要保持轻薄，只负责接收请求、参数校验、调用服务、返回结果，不承载核心业务判断。
2. 业务规则优先放在 `Service` 层，保证用例表达清楚，也方便单元测试。
3. 数据访问细节放在 `Repository`、`Mapper` 或持久层模块中，不要把 SQL 或查询细节泄露到控制层。
4. 请求 DTO、响应 DTO、持久化 Entity 要分开，避免直接把数据库实体暴露成接口契约。
5. 结构性校验优先通过注解完成，业务性校验放在服务层处理。
6. 事务边界定义在服务层。
7. 异常处理应统一走全局异常处理器，保证返回结构一致。
8. 类名和方法名要体现业务语义，尽量少用含义模糊的通用名字。
9. 日志要服务于排障和运维，但不能打印密码、令牌、隐私信息等敏感数据。
10. 对外接口如果已经被使用，默认要考虑兼容性，除非本次改动就是明确版本升级。

## 接口设计建议

- 命名尽量体现资源语义。
- HTTP 方法要与操作意图一致。
- 返回状态码要清晰准确。
- 列表、详情、新增、更新、删除等接口风格尽量统一。
- 分页接口的请求和返回结构要标准化。

## DTO 与 Entity 边界

- 请求 DTO 应描述客户端传入的数据，不应直接等同于数据库表结构。
- 响应 DTO 只返回前端真正需要的字段。
- Entity 保持面向持久化。
- 如果跨层传递 Entity 会让语义变差，应改为 DTO 或更明确的领域对象。

## 校验与异常处理

- 必填项、长度、格式等校验放在请求 DTO。
- 跨字段和业务规则校验放在服务层。
- 返回统一、稳定的错误码和可理解的错误信息。
- 不要把框架原始异常、SQL 异常或堆栈信息直接暴露给前端。

## 事务处理

- 涉及多表写入或需要原子性的服务方法，应显式考虑事务。
- 事务方法保持聚焦，不要又长又杂。
- 对事务中的外部调用要谨慎。
- 支付、回调、消息消费等容易重试的流程，要考虑幂等性。

## 日志与可观测性

- 日志中尽量带上可追踪标识，例如请求 ID、用户 ID、订单 ID、任务 ID。
- 重点记录状态变化和失败原因，不要打印无意义的流水账。
- 要区分“业务拒绝”和“系统异常”。

## 与 MySQL 和 Redis 的配合意识

使用这个 skill 时，同时要关注：

- 当前查询是否对 MySQL 索引友好
- 当前操作是否需要读取缓存或失效缓存
- 写操作之后是否需要缓存删除、延迟重建或幂等控制

## 使用时的检查清单

使用这个 skill 时，重点检查：

- `Controller` 是否过重
- DTO 和 Entity 是否混用
- 结构性校验和业务校验是否分层合理
- 事务边界是否清晰
- 异常处理是否统一
- 这个接口在半年后是否仍然容易理解
- 数据库和缓存之间是否保持一致性

## 期望结果

- 分层边界更清晰
- 接口契约更稳定
- 参数校验和事务处理更安全
- Spring Boot 服务更容易维护
- 与 MySQL、Redis 的协作更合理
