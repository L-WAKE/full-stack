# Redis 缓存模式规范

## 作用

当你在 Spring Boot 服务中引入或评审 Redis 用法时，使用这个 skill，尤其适合缓存、幂等控制、限流、轻量级协调等场景。

## 适用场景

- 设计缓存读写流程
- 检查 key 命名和 TTL 策略
- 防止缓存穿透、击穿、雪崩
- 增加幂等控制、分布式锁、计数器
- 保证 Redis 使用方式和 MySQL 主数据逻辑一致

## 核心规则

1. 除非业务本身就是 Redis 原生场景，否则默认 MySQL 才是最终数据源。
2. 一个功能内要明确选择一种缓存模式，通常优先使用 cache-aside，不要同一模块混多套方案。
3. key 命名规范要清晰，包含业务前缀和必要标识。
4. 每个缓存项都要有明确 TTL，或者明确说明为什么要长期存在。
5. 缓存失效策略要在设计写流程时一起考虑，不要等线上出错后再补。
6. 高度易变、用户专属且复用价值低、或不适合暴露的数据，不要轻易缓存。
7. 对热点 key 和高并发重建场景，要提前考虑防击穿措施。
8. 分布式锁要谨慎使用，必须有超时、失败处理和释放策略。
9. Redis 中存储的数据尽量保持轻量、可版本化。
10. 不要让 Redis 逐步演变成一个没人治理的“第二数据库”。

## 推荐模式

### Cache-Aside

- 先查 Redis
- 缓存未命中时查 MySQL
- 查到后回写 Redis，并设置 TTL
- 写操作后执行缓存删除或刷新

### 幂等控制

- 对创建、回调、重试型请求，可以使用短期 key 做去重
- 必要时区分处理中、成功、失败等状态

### 限流与计数

- 利用 Redis 原子操作处理短时间窗口计数
- 要明确窗口重置规则和失败兜底逻辑

## Key 设计建议

推荐使用可读性较好的 key，例如：

- `user:profile:{userId}`
- `product:detail:{productId}`
- `order:create:idempotency:{requestId}`

规则建议：

- 前缀按业务域统一
- 只保留查找所需的关键标识
- 避免随意拼接、语义不清的 key
- 当缓存结构变化较大时，可以通过版本号区分 key

## TTL 建议

- 高频变化或风险敏感的数据，TTL 应更短
- 读多写少的基础数据，可以适当放宽 TTL
- 大量 key 同时创建时，TTL 最好加入随机抖动，降低同时过期风险
- 每类缓存最好都能说明为什么设置成当前时长

## 常见风险点

- 缓存穿透：查询不存在的数据导致反复打到数据库
- 缓存击穿：热点 key 失效后大量请求同时重建
- 缓存雪崩：大量 key 在同一时间集中失效
- 数据更新后缓存未及时失效，导致脏数据
- value 过大或 key 数量失控

## 使用时的检查清单

使用这个 skill 时，重点检查：

- Redis 是否真的是这个问题的合适方案
- 当前采用的是哪种缓存模式
- 是否有明确的缓存失效或刷新策略
- key 命名和 TTL 是否统一
- 并发缓存未命中时会发生什么
- 是否仍然保证 MySQL 作为主数据源的一致性预期

## 期望结果

- 缓存行为更稳定、可预测
- 数据库压力更低
- 缓存相关故障更少
- Redis 的使用边界更清晰、更容易治理
