# Vite 项目规范

## 作用

当你在 Vue 3 + Vite 项目中处理工程结构、构建配置、环境变量、性能优化和部署相关问题时，使用这个 skill。

## 适用场景

- 新增或调整前端目录结构
- 修改 `vite.config.*`
- 检查环境变量使用方式
- 优化打包结构与路由加载方式
- 为项目上线或部署做准备

## 核心规则

1. `Vite` 配置尽量保持精简和明确，不要无目的地堆插件。
2. 路径别名要保持一致，`vite.config.*`、`tsconfig.*`、测试配置、lint 配置都应统一。
3. 当项目规模变大时，优先按功能边界拆分代码，而不是所有代码都堆在一起。
4. 环境变量统一使用 `VITE_` 前缀，并尽量通过一个配置层集中读取，不要散落在各个组件里直接访问。
5. 页面级路由优先使用懒加载，除非确实有首屏渲染需求必须提前引入。
6. 资源文件、图标、mock 数据也要有明确目录，不要越积越乱。
7. 网络请求优先集中到 `api` 模块，不要把请求逻辑散落在很多页面和组件内部。
8. 引入 UI 库、图表库、编辑器、SDK 时，要关注打包体积和首屏性能影响。

## 推荐的前端目录结构

在适合当前项目的前提下，建议逐步靠近下面这种结构：

- `src/api`：接口请求封装
- `src/assets`：静态资源
- `src/components`：通用展示组件
- `src/composables`：可复用的 Vue 逻辑
- `src/features`：按业务功能划分的模块
- `src/router`：路由定义与守卫
- `src/stores`：Pinia 状态管理
- `src/utils`：与框架无关的工具方法
- `src/views`：页面级视图组件

## 环境变量约定

- 只有适合暴露到浏览器端的值，才允许放到 `VITE_` 环境变量中。
- 多处会用到的环境变量，建议集中到一个配置文件统一读取。
- 必填环境变量在项目启动阶段最好就进行校验。
- 变量命名尽量明确，例如 `VITE_API_BASE_URL`，不要使用过于模糊的名字。

## 路由与加载建议

- 页面级组件优先使用动态导入实现按路由拆包。
- 路由守卫要保持职责单一、可组合，不要越来越臃肿。
- 不要让每个页面各自处理一套完全不同的鉴权、布局和数据请求流程。
- 预加载只在明确对用户体验有帮助时才使用。

## 性能优化建议

- 引入新依赖前，先评估体积和运行时成本。
- 优先选择支持 tree-shaking 的库。
- 重型依赖不要默认放在应用入口加载，尽量按功能拆分。
- 对图表库、富文本编辑器、代码高亮库、时间处理库要特别谨慎。
- 利用好 Vite 默认的构建产物哈希和浏览器缓存能力。

## API 接入建议

- `baseURL`、鉴权头、超时、错误处理尽量统一管理。
- `fetch` 或 `axios` 的底层调用不要散落在大量视图文件中。
- 后端返回的数据应先整理成前端更容易使用的结构，再继续向下传递。

## 使用时的检查清单

使用这个 skill 时，重点检查：

- 当前目录结构是否还能反映产品的功能边界
- 环境变量是否安全、命名是否统一
- 路由是否按页面粒度做了懒加载
- 新依赖是否显著增加了打包成本
- API 请求是否集中管理并做了统一处理
- `Vite` 配置在变更后是否仍然容易理解

## 期望结果

- 项目结构更清晰
- 客户端配置更安全
- 路由拆包和按需加载更合理
- API 调用方式更统一
- Vite 工程更容易长期维护
