# Vue3 组件开发规范

## 作用

当你在当前项目中编写、修改、重构 Vue 3 组件时，使用这个 skill。它适用于基于 Composition API 和 `script setup` 的开发方式。

## 适用场景

- 新建 Vue 组件
- 重构已有组件
- 检查组件职责是否清晰
- 提取 composable
- 统一 `props`、`emits`、`slots` 和 `v-model` 的写法
- 提升组件可维护性、可读性和可测试性

## 核心规则

1. 单文件组件优先使用 `script setup`，除非当前文件已经明确遵循另一套团队约定。
2. 一个组件尽量只负责一个明确的界面职责。若一个文件同时混合页面编排、数据请求、展示细节，应考虑拆分。
3. 当某段状态逻辑会被多个组件复用，或单个组件已经变得难以阅读时，应提取为 composable。
4. `props` 要保持明确、精简、类型清晰。不要为了省事传入一个过大的对象。
5. 优先保持单向数据流。父组件拥有的数据，应通过 `emit` 或 `v-model` 回传更新，不要在子组件中直接改嵌套属性。
6. 派生状态优先使用 `computed`，避免把可以推导出的值再次存进本地状态。
7. `watch` 主要用于副作用，例如同步接口、路由、缓存、第三方库，不要把它当成普通数据推导工具。
8. 模板中避免过深的条件分支。复杂逻辑应转移到 `computed` 或更小的展示组件中。
9. 纯展示组件中不要塞入过多业务逻辑。
10. 尽量沿用项目已有的 UI 组件库和交互模式，不要在同一个项目里再引入第二套完全不同的风格。

## 推荐的组件内部结构

组件内部建议按下面顺序组织：

1. `import`
2. `props` 和 `emits`
3. 路由、状态管理相关 hooks
4. 响应式状态
5. 计算属性
6. 生命周期
7. 事件处理函数
8. 副作用 `watch`
9. 必要时暴露的方法

## Props、Emits 与 v-model 约定

- `props` 需要有明确类型，必要时提供默认值。
- `emits` 命名要贴近用户意图，例如 `save`、`cancel`、`submit`、`update:modelValue`。
- `v-model` 适合表单型、受控型交互，不要把所有子传父通信都强行写成 `v-model`。
- 如果一个组件存在多个受控值，使用具名 `v-model` 时要有明确意图，必要时可加简短注释说明。

## Composable 提取原则

在下面这些情况下，优先考虑提取 composable：

- 一段逻辑会被多个组件复用
- 异步状态管理已经影响组件主体可读性
- 路由、筛选、表格、查询参数等逻辑逐渐变重

提取 composable 时遵循这些规则：

- 统一命名为 `useXxx`
- 返回值保持精简、稳定
- 尽量隐藏请求或底层实现细节
- 异步场景要暴露 `loading`、`error` 等必要状态
- DOM `ref` 一般留在组件内部，除非 composable 本身就是为 DOM/交互封装而写

## 模板编写建议

- 优先可读性，不要为了省几行代码写过于花哨的表达式。
- 避免大量内联三元表达式连在一起。
- `v-for` 的 `key` 必须稳定，除非列表是完全静态且只追加，否则不要用数组下标。
- 当组件扩展需求很多时，优先考虑 `slot`，不要不停堆布尔型 `props`。

## 状态与数据请求建议

- 服务端数据的归属应尽量靠近页面级组件或功能容器组件。
- 传给子组件的数据尽量是已经整理过的，不要让每个子组件都理解后端原始返回格式。
- 后端响应在进入组件树之前，最好先做一次结构整理。
- 避免因为生命周期和 `watch` 同时存在，造成重复请求。

## 使用时的检查清单

使用这个 skill 时，重点检查：

- 当前组件是不是承担了过多职责
- 有没有本地状态其实可以改成 `computed`
- `props` 是否过宽、过松
- `watch` 是否被用来做本该由 `computed` 完成的事情
- 重复逻辑是否应该提取为 composable
- 模板是否需要“脑内模拟很多分支”才能看懂
- 组件对外暴露的 API 是否符合父组件的自然使用方式

## 期望结果

- 组件更小、更清晰
- 父子组件边界更明确
- 可测试性和可维护性更好
- composable 复用能力更强
- 重复状态和意外副作用更少
