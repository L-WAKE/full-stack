# 前端样式规范

## 作用

当你在当前项目中编写、修改、重构前端样式时，使用这个 skill。它适用于 Vue 3 + Vite 项目中的 `CSS`、`SCSS`、组件局部样式、页面样式和主题变量管理。

## 适用场景

- 新增页面或组件样式
- 重构已有样式结构
- 统一类名命名方式
- 规范颜色、间距、字体、圆角、阴影等设计变量
- 优化响应式布局
- 排查样式污染、优先级冲突和重复样式

## 核心规则

1. 样式要服务于组件和页面结构，不要堆无语义、难维护的类名。
2. 优先通过设计变量统一颜色、字号、间距、圆角、阴影、层级，不要在各处直接写零散魔法值。
3. 组件样式默认应尽量局部化，避免无边界地污染全局。
4. 样式命名要清晰稳定，看到类名就能大致判断用途。
5. 选择器层级不要过深，避免为了覆盖而不断提高优先级。
6. 相同视觉模式应复用，不要复制粘贴多份近似样式。
7. 布局优先使用 `flex` 和 `grid`，不要依赖过多历史性 hack。
8. 响应式设计要从常见屏宽和内容约束出发，不要只追求“缩小不乱”。
9. 状态样式要完整考虑，例如默认、悬停、激活、禁用、错误、加载态。
10. 保持样式与现有项目视觉语言一致，除非任务明确要求重做视觉体系。

## 命名建议

- 通用推荐使用有语义的类名，例如 `user-card`、`toolbar-actions`、`filter-panel`。
- 组件内部可使用较轻量的块结构命名，例如：
  - `order-form`
  - `order-form__header`
  - `order-form__actions`
  - `is-disabled`
- 不要使用纯展示结果命名，例如 `red-text`、`mt12`、`big-box`，除非项目本身已经采用工具类体系。

## 变量规范

推荐把视觉变量统一管理，例如：

- 颜色变量：主色、成功色、警告色、危险色、文字色、边框色、背景色
- 间距变量：`4 / 8 / 12 / 16 / 24 / 32` 这类有节奏的间距体系
- 圆角变量：小、中、大
- 阴影变量：轻、中、重
- 层级变量：浮层、弹窗、通知

如果项目支持主题，变量应优先通过 `CSS Variables` 暴露，例如：

- `--color-primary`
- `--text-primary`
- `--bg-page`
- `--border-default`
- `--radius-md`

## 组件样式建议

- Vue 单文件组件优先让样式和组件共置，方便理解和维护。
- 如果一个样式会被多个组件复用，应抽到公共样式层，而不是复制多份。
- `scoped` 适合大多数组件，但如果需要统一主题或基础原子规则，可放到全局样式中集中维护。
- 不要过度依赖 `:deep()`，只有在第三方组件样式覆盖确实必要时再使用。

## 布局建议

- 一维排列优先使用 `flex`
- 二维区域布局优先使用 `grid`
- 表单、卡片区块、工具栏、列表区要保持统一间距节奏
- 页面最大宽度、内容留白、主副栏比例要有明确约束
- 避免随意写死高度，尤其是内容动态变化的区域

## 响应式建议

- 至少考虑桌面端、平板端、小屏移动端三个层级
- 优先先处理内容断裂、换行、遮挡、溢出问题
- 弹窗、表格、表单在小屏场景下要重点检查
- 当某块内容在小屏下无法保持可用性时，应改布局，而不是单纯缩小字号

## 状态与交互样式

- 按钮、输入框、标签、卡片等常见组件都要考虑交互状态
- 可点击区域要有明确反馈，不能只有鼠标移入才勉强区分
- 禁用态要做到“可识别但不过度抢眼”
- 错误态、校验提示、空状态、加载态要保持统一风格

## 使用时的检查清单

使用这个 skill 时，重点检查：

- 是否存在大量重复样式
- 是否有太多魔法值直接写在组件里
- 类名是否清晰、稳定、可维护
- 选择器层级是否过深
- 是否出现了全局污染或覆盖链失控
- 响应式布局是否真的可用，而不仅仅是没有报错
- 是否已经统一使用颜色、间距、字号、圆角等变量

## 期望结果

- 样式结构更清晰
- 视觉规则更统一
- 组件样式边界更明确
- 响应式表现更稳定
- 后续维护和扩展成本更低
